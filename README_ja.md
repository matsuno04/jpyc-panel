# JPYC オンチェーン・パネルデータ基盤

新JPYC(電子決済手段、`0xE7C3D8C9a439feDe00D2600032D5dB0Be71C3c29`、Ethereum / Polygon / Avalanche / Kaia 共通)の**全Transferイベント**をパブリックRPCから取得し、卒業研究(拡散プロセスの実証分析)に直接使える日次パネルデータセットを構築するツール一式。APIキー・課金は一切不要。

> 注意: 前払式JPYC(JPYC Prepaid、旧v1)は**別コントラクト**。この基盤は電子決済手段版のみを対象とする。両者を比較したい場合は config.py にコントラクトを追加すれば同じ仕組みで収集できる。

---

## 1. ファイル構成

| ファイル | 役割 |
|---|---|
| `config.py` | チェーン設定・RPCエンドポイント・閾値・バケット定義。**研究上のパラメータは全部ここ** |
| `rpc.py` | JSON-RPCクライアント(エンドポイント自動切替・リトライ) |
| `collector.py` | 全Transferログの取得(チェックポイント付き、中断・再開可) |
| `panel_builder.py` | 生イベント → 日次パネル・アドレス台帳・コホート保持・スナップショット |
| `analyze.py` | 標準図表8枚をPNG出力 |
| `known_addresses.csv` | 運営・CEX等の手動ラベル(除外リスト) |
| `dune_queries.sql` | Duneでの検証用クエリ3本 |
| `JPYC_panel_colab.ipynb` | Colab実行用ノートブック |
| `make_synthetic.py` | 合成データ生成(動作確認用。実運用では不要) |

## 2. 使い方(最短ルート)

```bash
pip install pyarrow            # Colabはこれだけ。pandas等は既存
python collector.py avalanche  # 軽いチェーンで動作確認
python collector.py            # 全4チェーン(Polygonが一番重い)
python panel_builder.py        # パネル構築(チェーン別 + combined)
python analyze.py combined     # 図表生成
```

- **中断・再開**: `collector.py` は `data/checkpoint_{chain}.json` に進捗を保存する。Colabが切れても同じコマンドで続きから走る。Googleドライブをマウントして作業ディレクトリをドライブ上に置くこと(ノートブック参照)。
- **更新**: 後日もう一度 `collector.py` を実行すれば前回の続き(新しいブロック)だけ取得する。日次パネルを最新化するには builder → analyze を再実行。
- **やり直し**: `data/checkpoint_*.json` と `data/shards_*` を消せば最初から。
- **完全性チェック**: builder 実行時に「⚠ 負残高アドレス」警告が出たら**取得漏れ**(ERC-20で残高は負にならないため)。そのチェーンの checkpoint を消して収集し直す。
- **総量検証**: 最新日の `circulating_supply` を JPYC Info や explorer の totalSupply、`dune_queries.sql` の結果と突き合わせる。誤差ゼロで一致するのが正常(丸め以外)。

## 3. 出力ファイルの読み方

scope は `ethereum / polygon / avalanche / kaia / combined`。**combined はアドレス単位でチェーン横断合算したもの**——EVM系では同一アドレス=同一の鍵保有者である蓋然性が高いので、これが「ユニーク保有者数」の最良の近似になる。チェーン別の単純合計は同一人物を重複カウントするので、公表値(例: JPYC Infoのホルダー数)と比較する際はどちらの定義かを必ず確認すること。**論文の112,090→60,587という数字がどちらの定義だったかも、この基盤で再現して確かめられる。**

### 3.1 `daily_panel_{scope}.csv` — 本体

| 列 | 定義 | 研究上の意味・読み方 |
|---|---|---|
| `transfers` | その日のTransfer件数 | 活動量の最も生な指標。**イベントスタディの従属変数候補** |
| `mint_count / mint_volume` | 0x0からの発行(=JPYC EXでの発行) | **一次需要**。新規資金流入 |
| `burn_count / burn_volume` | 0x0への償還 | 出口。mintとの差の累積が流通量 |
| `p2p_volume` | 発行・償還以外の送金額 | **実際に「使われている」度合い**。mint/burnに対するp2pの比率が上がる=保有から利用への移行 |
| `unique_senders / receivers / active_addresses` | その日に送金/受取/どちらかをしたアドレス数 | DAU相当。holdersと違い「動いている」人数 |
| `new_addresses` | 初めて残高>0になったアドレス数 | **Bassモデルの採用者系列はこれの累積**。イベントスタディの従属変数の第一候補 |
| `zeroed_addresses` | 残高が>0→0になった数 | 離脱。new−zeroedが純増 |
| `resurrected_addresses` | 一度ゼロ化→再び>0 | 再利用。これが多い=決済利用(使い切って補充)の兆候でもある |
| `holders_gt0` | 残高>0のアドレス数 | 公表される「ホルダー数」に相当 |
| `holders_ge1/100/1000/10000` | 残高が各閾値(円)以上の数 | **ダスト感度分析**。下記4.1参照 |
| `circulating_supply` | 総流通量(除外アドレス除く残高合計) | JPYC Infoの流通量と照合 |
| `n_{bucket} / val_{bucket}` | 保有帯別のアドレス数/金額 | 帯: <1千 / 1千〜1万 / 1万〜10万 / 10万〜100万 / 100万円〜 |
| `share_holders_lt10k` | 1万円未満保有者の割合 | **論文の「82–98%」主張の直接検証**。時系列で見られるのがポイント |
| `gini / top10_share / top100_share` | 集中度 | 分散化の進行度。低下トレンド=一般層への広がりと整合 |
| `mean / median_balance` | 平均・中央値残高 | 平均≫中央値の乖離=分布の歪み |

### 3.2 `address_master_{scope}.csv` — アドレス台帳

1行=1アドレス。`first_seen`(獲得日) / `last_active` / 累計入出金 / 取引回数 / ユニーク相手方数 / mint受領・burn実行フラグ。

- **これが生存分析(Kaplan-Meier / Cox)の入力**。`first_seen`が観察開始、`balance==0`かつ`last_active`が離脱イベント、`balance>0`が打ち切り。
- **共変量が既に入っている**: 獲得時期(キャンペーン期ダミー)、mint受領(自分で発行した=能動的ユーザー)vs 受取のみ(エアドロップ/送金受領)、取引回数、相手方数。
- 将来のXデータ連携: ウォレットアドレスを公開しているXアカウントとの突合キーにもなる。

### 3.3 `cohort_retention_{scope}.csv` + ヒートマップ(図08)

獲得週コホート×経過週の保有継続率(残高≥`RETENTION_THRESHOLD`)。**「キャンペーン獲得層は定着しない」仮説の直接検証**。キャンペーン週のコホートだけ保持曲線が急落していれば仮説支持。行(コホート)ごとに横に読む: 右に行くほど暗くなる速度が離脱速度。

### 3.4 `flag_candidates_{scope}.csv` — 要ラベリング

ユニーク相手方数(degree)と取扱量の上位40アドレス。CEXホットウォレット・運営・決済コントラクトはここに浮上する。**explorerで確認して `known_addresses.csv` に `category=exclude` で追記 → builder再実行**、が運用ループ。除外前後の両方の数字を持っておくと、頑健性チェック(robustness)として論文に書ける。

### 3.5 `balances_latest_{scope}.csv`

最新の残高スナップショット(降順)。上位保有者の確認、保有分布の任意の再集計に。

## 4. 研究への接続(どの分析に何を使うか)

### 4.1 「112k→60k問題」の分解(最優先でやる価値がある)
図01(閾値バンド)を見る。**`holders_gt0`が急落しているのに`holders_ge1000`がほぼ平坦なら、減少の正体は「ダストの掃き出し」であって実ユーザーの離脱ではない**。逆に全バンドが同時に落ちていれば本当の離脱。さらに図02の`zeroed_addresses`のスパイク日と、ニュースパイプラインのイベント(キャンペーン終了日など)を突き合わせれば、機序まで特定できる。これだけで論文の5.2節の「仮説」を検証結果に格上げできる。

### 4.2 Bassモデル
`new_addresses`の累積(=累積採用者数。純減しない系列)に対してp(革新係数)・q(模倣係数)を推定。`scipy.optimize.curve_fit`で3パラメータ(m, p, q)を当てるだけ。チェーン別に推定して比較するのも面白い(Kaia=LINE系ユーザーは模倣係数が高い、等の仮説)。

### 4.3 イベントスタディ/中断時系列
従属変数: `new_addresses`(獲得)、`mint_volume`(資金流入)、`p2p_volume`(利用)。説明変数: ニュースDBのタグ付きイベント日ダミー。カウントデータなのでPoisson/負の二項回帰、曜日固定効果とトレンドを入れる。**「どの種類のイベントが獲得を生み、どれが利用を生むか」**が答えられる。

### 4.4 生存分析
3.2の通り。コホート(獲得週)を層別にしたKaplan-Meier曲線を並べるのが第一歩。

### 4.5 規制起因仮説のベースライン比較
「1万円未満82–98%は規制の帰結」と主張するには、**同じ手法を他トークン(例: Avalanche上のUSDC)に適用して分布を比較**する必要がある。config.pyのコントラクトアドレスを差し替えるだけで同じパイプラインが走る。他トークンでも小口が8–9割なら、規制ではなくトークン一般の性質。

## 5. 既知の限界(論文のlimitationsにそのまま書ける)

1. **アドレス≠人**。1人が複数アドレス、CEX内の多数ユーザーが1アドレス。除外リストと閾値感度で緩和はできるが解消はしない。
2. **combinedの前提**: 同一アドレス=同一主体はEVM系では強い仮定ではないが、スマートコントラクトウォレット等の例外はある。
3. **タイムスタンプはグリッド補間**(誤差±数分程度)。日次集計には影響しないが、時間単位の分析をするなら `ts_grid_step` を小さくする。
4. **CEX内部の移転は見えない**(オフチェーン)。取引所経由の保有・売買はmint/burn/入出金しか観測できない。
5. パブリックRPCはレート制限がある。Polygonの初回フルスキャンは数時間かかり得る。失敗しても再開できるので放置でよい。

## 6. カスタマイズ

- 閾値・バケット・保持判定: `config.py` の `DUST_THRESHOLDS / BALANCE_BUCKETS / RETENTION_THRESHOLD`
- RPCが不調: `config.py` の `rpcs` リストに [chainlist.org](https://chainlist.org) から生きているエンドポイントを足す(先頭が優先)
- 他トークンの比較収集: `JPYC_CONTRACT` を差し替え、`DATA_DIR/OUT_DIR` を変えて実行
